# HE Wiki Graph Pipeline - Backend Implementation Plan

## Goal

Parse the Obsidian wiki graph in `backend/he_graph/` (47 .md files across 7 categories), convert it to JSON, and load it into Neo4j so the frontend can display it as a standalone programme graph.

---

## 1. Context: How Existing Programmes Work

### Data Flow

```
PDF Work Programme
  -> Parser script (he_wp_parser.py)
  -> Grouped JSON (cluster_CL*.grouped.json)  +  Summaries JSON (destination_summaries_cl*.json)
  -> Cluster Builder (extends BaseClusterBuilder)
  -> Neo4j (via cluster_routes_factory endpoints)
  -> Frontend (via /prefix/nodes + /prefix/relationships)
```

### Neo4j Schema (Existing Clusters)

```
(:Cluster {id, name, source, type, summary})
  -[:HAS_DESTINATION]->
(:Destination {id, name, source, type, summary})
  -[:HAS_CALL]->
(:Call {id, name, source, type, ...50+ props})
```

- All nodes share: `id`, `name`, `source` (filter key), `type` (label for frontend)
- The `source` property (e.g. `"cluster_2"`) is used everywhere for scoped queries
- Routes are generated by `cluster_routes_factory.make_cluster_router()`, providing: `GET /nodes`, `GET /relationships`, `GET /node/{id}`, `POST /populate`, `DELETE /all`

### Key Difference

The Obsidian graph is **not hierarchical** (Cluster -> Destination -> Call). It is a **flat network** of 47 entities connected by ~400 wikilinks. It needs a different builder, but the same route pattern.

## 2. Obsidian Graph Structure

### Folder Layout (7 categories, 47 entity pages)

```
backend/he_graph/
  strategies/      (9 files)  - Horizon Europe, Green Deal, Clean Industrial Deal, ...
  clusters/        (6 files)  - Cluster 1-6
  institutions/    (9 files)  - EC, ERC, EIC, EIT, JRC, MSCA, ...
  policies/        (9 files)  - Open Science, Gender Equality, DNSH, ...
  research_themes/ (8 files)  - AI, Climate Neutrality, Biodiversity, ...
  topics/          (8 files)  - EU Missions, Lump Sum Funding, ...
  syntheses/       (5 files)  - Strategic maps, ecosystem analyses
  index.md                    - Master index (metadata, not a graph node)
  log.md                      - Ingestion log (metadata, not a graph node)
```

### Frontmatter Schema (YAML)

Every entity .md file has:

```yaml
title: "Horizon Europe"
category: strategy              # one of: strategy, cluster, institution, policy, research_theme, topic, synthesis
source_documents:               # list of PDF filenames
  - programme-guide_horizon_en.pdf
related_nodes:                  # explicit named relationships (subset of all wikilinks)
  - European Commission
  - European Research Council
keywords:                       # list of keywords
  - framework programme
  - R&I
aliases:                        # alternative names
  - HE
  - HORIZON
status: active
last_updated: 2026-05-11
```

### Body Content

- Markdown sections (Overview, Strategic Role, Connections, Sources, etc.)
- `[[wikilinks]]` embedded in body text pointing to other entities
- The union of `related_nodes` + body wikilinks forms the complete edge set

## 3. Pipeline Design

### 3.1 Step 1: Parse .md Files -> Intermediate JSON

**Script**: `backend/routes/new_pipeline/he_wiki/parse_obsidian.py`

**Input**: `backend/he_graph/**/*.md` (skip `index.md` and `log.md`)

**Logic**:
1. Walk the `he_graph/` directory, reading each `.md` file
2. Parse YAML frontmatter (use `python-frontmatter` or manual `---` split + `yaml.safe_load`)
3. Extract all `[[wikilinks]]` from the body text using regex: `\[\[([^\]]+)\]\]`
4. For each file, build a node dict and a relationships list
5. Generate a node `id` by slugifying the filename (e.g. `"Horizon Europe"` -> `"horizon-europe"`)
6. Build a title-to-id lookup map so wikilink targets can be resolved

**Output**: Two JSON files written to `backend/routes/new_pipeline/he_wiki/output/`

#### `he_wiki_nodes.json`

```json
[
  {
    "id": "horizon-europe",
    "name": "Horizon Europe",
    "type": "strategy",
    "source": "he_wiki",
    "category": "strategy",
    "summary": "<first paragraph of body or Overview section>",
    "keywords": ["framework programme", "R&I", "three pillars"],
    "aliases": ["HE", "HORIZON"],
    "source_documents": ["programme-guide_horizon_en.pdf"],
    "body": "<full markdown body text>",
    "status": "active"
  }
]
```

#### `he_wiki_relationships.json`

```json
[
  {
    "source": "horizon-europe",
    "target": "european-commission",
    "type": "RELATES_TO",
    "origin": "frontmatter"
  },
  {
    "source": "horizon-europe",
    "target": "european-research-council",
    "type": "WIKI_LINK",
    "origin": "body"
  }
]
```

**Relationship type rules**:
- `related_nodes` from frontmatter -> `RELATES_TO` (explicit, curated)
- `[[wikilinks]]` in body not already in `related_nodes` -> `WIKI_LINK` (contextual)
- Both are deduplicated (one edge per source-target pair; frontmatter takes precedence)

### 3.2 Step 2: Neo4j Builder

**Script**: `backend/routes/new_pipeline/he_wiki/he_wiki_builder.py`

This does **not** extend `BaseClusterBuilder` (which assumes Cluster -> Destination -> Call hierarchy). Instead, it is a standalone builder class following the same conventions.

```python
class HEWikiGraphBuilder:
    SOURCE_TAG = "he_wiki"

    def __init__(self, preview=False):
        self.preview = preview

    def create_graph_from_files(self, nodes_path: str, relationships_path: str):
        """
        1. Load he_wiki_nodes.json and he_wiki_relationships.json
        2. For each node: MERGE (:HEWikiNode {id}) SET properties
        3. For each relationship: MATCH + MERGE edge
        """
```

**Neo4j Schema**:

```
(:HEWikiNode {
    id,                    # "horizon-europe"
    name,                  # "Horizon Europe"
    type,                  # "strategy" (= category)
    source,                # "he_wiki" (constant, for scoped queries)
    category,              # "strategy" | "cluster" | "institution" | "policy" | "research_theme" | "topic" | "synthesis"
    summary,               # first paragraph / Overview section
    keywords,              # string[] from frontmatter
    aliases,               # string[] from frontmatter
    source_documents,      # string[] from frontmatter
    body,                  # full markdown body (for detail panel / chatbot)
    status                 # "active"
})
```

Relationships:
```
(:HEWikiNode)-[:RELATES_TO]->(:HEWikiNode)   # from frontmatter related_nodes
(:HEWikiNode)-[:WIKI_LINK]->(:HEWikiNode)    # from body [[wikilinks]]
```

**Why a single label `HEWikiNode` instead of per-category labels**: The `source` filter (`WHERE n.source = 'he_wiki'`) already scopes all queries, matching the existing pattern. The `category` property enables frontend coloring/grouping. This avoids needing 7 different label types in Cypher queries and keeps the route factory compatible.

### 3.3 Step 3: API Routes

**Script**: `backend/routes/new_pipeline/he_wiki/he_wiki_routes.py`

Use a custom router (not `make_cluster_router`, since populate takes two different JSON files and the node label differs):

```python
router = APIRouter(prefix="/hewiki", tags=["HE Wiki Knowledge Graph"])
```

**Endpoints** (matching existing pattern):

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/hewiki/nodes` | All HEWikiNode nodes (source=he_wiki) |
| `GET` | `/hewiki/relationships` | All relationships between he_wiki nodes |
| `GET` | `/hewiki/node/{id}` | Single node by id |
| `POST` | `/hewiki/populate` | Run builder from JSON files |
| `DELETE` | `/hewiki/all` | Delete all he_wiki nodes + relationships |

### 3.4 Step 4: Register in main.py

```python
from routes.new_pipeline.he_wiki.he_wiki_routes import router as hewiki_router
app.include_router(hewiki_router)
```

## 5. Implementation Order

1. **`parse_obsidian.py`** - Can be run standalone (`python parse_obsidian.py`) to generate the JSON files. No Neo4j dependency.
2. **`he_wiki_builder.py`** - Reads the JSON files and writes to Neo4j. Test with `preview=True` first.
3. **`he_wiki_routes.py`** - Wire up FastAPI endpoints.
4. **Register in `main.py`** - One import + one `app.include_router()` call.
5. **Test** - Hit `POST /hewiki/populate`, then `GET /hewiki/nodes` and `GET /hewiki/relationships` to verify.

---

## 6. Design Decisions

| Decision | Rationale |
|----------|-----------|
| Custom builder, not `BaseClusterBuilder` | The wiki graph is a flat network, not a Cluster->Destination->Call hierarchy |
| Single `HEWikiNode` label | Matches the `source`-based filtering pattern used everywhere; `category` property handles frontend grouping |
| Two relationship types (`RELATES_TO` / `WIKI_LINK`) | Distinguishes curated frontmatter links from contextual body mentions; frontend can filter or weight differently |
| Summary = first paragraph of Overview | Consistent with how destination summaries work in existing pipeline |
| Body stored as property | Enables detail panel display and chatbot context without a second query |
| Slugified filename as `id` | Deterministic, human-readable, matches wikilink resolution |
| JSON intermediate files | Decouples parsing from loading; files can be inspected/edited; matches existing pipeline pattern |
